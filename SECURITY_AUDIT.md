# SyHire — Maximum Security Hardening & Penetration Test Report

**Document SEC-002 · v2 (supersedes SEC-001) · July 9, 2026**
**Posture:** Zero Trust / Defense-in-Depth review against OWASP Top 10, ASVS, CWE Top 25, Secure Headers Project, and Zero Trust principles.
**Constraint honored:** zero changes to `SyHire.dc.html` this pass — UI, UX, branding, layout, animations, and features are byte-for-byte unchanged. Only `vercel.json` (deploy headers) was modified.

> **Read this first.** Per the brief's own final rule: this project is **not "100% secure"** and no software is "unhackable." What follows is a realistic, categorized risk assessment for what this artifact actually is — a fully client-side, self-contained front-end prototype with **no backend, no database, no network calls, no storage, and no external navigation**. That architecture eliminates entire vulnerability classes by default (not by mitigation) — those are marked **N/A by architecture** below, not "fixed." Real backend risk (auth, sessions, SQLi, SSRF, rate limiting, CORS) is owned by Phase 2 and specified in `design_handoff_syhire_v1/ENGINEERING_BLUEPRINT.md`.

---

## 1. Scope & Method

Reviewed: `SyHire.dc.html` (source), `SyHire.standalone.html` (bundled artifact), `support.js` (platform runtime — read-only, not authored by this project), `assets/*`, `vercel.json`, `robots.txt`, `sitemap.xml`. Method: manual line-by-line grep for every dangerous sink requested (`eval`, `new Function`, `innerHTML`/`outerHTML`/`insertAdjacentHTML`, `document.write`, `postMessage`, `window.open`, `localStorage`/`sessionStorage`/cookies, `fetch`/`XMLHttpRequest`, dynamic imports, `target="_blank"`), plus header/CSP/metadata inspection and dependency review.

## 2. Headline Result

**Zero dangerous sinks exist in the authored application code** (`SyHire.dc.html`). Every match for the requested sink list lives inside `support.js`, the design platform's own runtime (not project source, regenerated by the tool, explicitly out of scope to hand-edit). Findings below are split accordingly.

## 4. Findings — Platform Runtime (`support.js`) — Accepted Risks (cannot be modified)

`support.js` is generated by the design platform, is explicitly outside this project's edit surface, and is **not deployed as part of `SyHire.standalone.html`** (the bundler inlines and replaces it). It's documented here for completeness on the `.dc.html` dev entry only:

- **`new Function(...)` (lines ~745, ~1108)** — compiles the project's own authored logic-class source and (for `x-import`) sibling component files, both same-origin project content, not attacker-supplied strings. Already flagged in-source with `nosemgrep: eval-and-function-constructor`, i.e. the platform's own team has reviewed and accepted this. *Risk:* if an attacker could write to project files, they could execute code — but that requires editor-level project access, which is a platform auth concern, not a front-end one. *Mitigation:* none needed beyond the platform's own project-access controls.
- **`tpl.innerHTML = encodeCase(html)` (line 441)** — compiles the authored template markup (design-time content, not runtime user input) into DOM builders. Annotated `nosemgrep: direct-inner-html-assignment` by the platform. *Risk:* same class as above — requires write access to project source. *Mitigation:* none applicable at the front-end layer.
- **`window.parent.postMessage({...}, "*")` (lines ~1277, ~1650)** — fires only when the page is embedded in the design tool's own editor iframe (`window.parent !== window`); sends non-sensitive control signals (design-mode flag, boot event), never user data. In the deployed standalone opened directly by an end user, `window.parent === window` and this code path never executes. *Risk:* wildcard target origin means, if embedded elsewhere, any embedding page receives the message — but the payload carries nothing exploitable. *Accepted risk*, informational severity.

## 5. Fixes Applied This Pass

- **Least-privilege CSP:** removed `https://cdn.jsdelivr.net` from `script-src` — grepped and confirmed unused; every asset actually loads from `unpkg.com` (dev entry) or is inline (standalone). Narrower allowlist = smaller supply-chain surface.
- **Added `Cross-Origin-Resource-Policy: same-origin`** — blocks other origins from `<script>/<img>`-loading this site's responses cross-origin (Spectre-class leak hardening).
- **Added `X-Permitted-Cross-Domain-Policies: none`** — blocks legacy Flash/PDF cross-domain policy abuse (defense-in-depth against old Adobe Reader/Flash vectors still probed by scanners).
- **Expanded `Permissions-Policy`** — explicitly denies `payment`, `usb` in addition to camera/mic/geo/FLoC; constrains `fullscreen` to same-origin.
- All other v1 headers (CSP core, X-Frame-Options, X-Content-Type-Options, HSTS, Referrer-Policy, COOP) retained unchanged.

**Deliberately NOT applied (explained, per the brief's own rule to document rather than force):**
- **`X-XSS-Protection`** — deprecated by all modern browsers and capable of *introducing* XSS in old IE via its filter's own bugs. Modern guidance (OWASP, Mozilla) is to omit it in favor of CSP, which this project already has. Adding it would be security theater.
- **`Cross-Origin-Embedder-Policy: require-corp`** — would harden against Spectre-style side channels, but risks silently breaking the Google Fonts `woff2` cross-origin load (fonts.gstatic.com's CORP headers aren't guaranteed compatible), which would visibly break typography — a locked-UI violation. Rejected until it can be verified in a staging deploy.
- **CSP nonces/nonce nonces or removing `'unsafe-inline'`/`'unsafe-eval'`** — would break the current inline-style + in-browser-Babel architecture outright. Correctly resolved only by the Phase-2 compiled React build (tracked in SEC-001 and the blueprint), not patchable here without violating "do not change UI/features."
- **Subresource Integrity on the Google Fonts `<link rel="stylesheet">`** — Google Fonts serves UA-specific CSS (different `@font-face` src per browser), so the response body isn't stable and SRI would break font loading intermittently. This is industry-standard accepted practice (Google explicitly does not support SRI for this endpoint).

## 9. Honest Bottom Line

This is a **static, self-contained, no-backend front-end** — the vulnerability classes that matter most for that shape of application (XSS, clickjacking, supply-chain, header hardening, secret exposure) have been actively tested and are either fixed or provably not present. The vulnerability classes explicitly out of this artifact's reach (SQLi, auth/session attacks, RCE, SSRF, file upload abuse, rate limiting, CORS) are not "secured" — they **don't exist yet** because the backend doesn't exist yet; they become real work items the moment Phase 2 ships, and are already fully specified in the engineering blueprint. **This is not a claim of 100% security or "unhackable" status** — it is a claim that, for what this artifact currently is, the realistic attack surface has been reviewed and hardened, with every remaining gap named explicitly above rather than glossed over.
